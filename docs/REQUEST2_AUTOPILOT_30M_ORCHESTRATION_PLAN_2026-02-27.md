## 5) 실패 처리 트리 (10분 내 강제)

1. 실패 감지 즉시 `Fail Ticket` 발행 (로그 ID, 재현 절차, 마지막 성공 SHA 포함).
2. 원인 4분류 고정: `Code`, `Environment`, `Data`, `Procedure`.
3. 10분 내 담당 재할당:
   - Code -> Dev lane 직접 핫픽스
   - Environment -> Ops lane 실행환경 복구
   - Data -> 데이터/스키마 복구 + 마이그레이션 검증
   - Procedure -> 실행순서/게이트 누락 보정
4. 재실행 후 동일 케이스 2회 연속 성공 전에는 완료 금지.

|---|---|---|

| G4 반영 | `main`에 merge/push 확인 | `main` SHA, push 시간 |

## 7) Part2 반복 운영 루프 (Part1 완료 이후)

1. Stabilization/Optimization(2.1)과 Enhancement(2.2)를 분리된 lane으로 병렬 운영.
2. 각 사이클 시작 시 backlog를 `Critical > High > Medium > Low`로 재정렬.
3. 코드 반영 없는 문서 갱신은 상태를 `In Progress`로만 인정.
4. 각 사이클 종료 시 다음 사이클용 후속 태스크 1건 이상 생성.

## 9) 산출물 체크리스트 (문서-only 완료 금지 장치)

- 코드: 기능 반영 커밋 SHA(또는 PR) 최소 1건 이상
- 검증: 테스트/재실행 로그 ID + QA 판정 리포트
- 반영: `main` push 확인 SHA/시간
- 운영: 30분 사이클 로그(진행률, 병목시간, 재시도횟수)
- 후속: 다음 사이클 할당 태스크와 담당자

위 5종 세트 중 하나라도 누락 시 해당 사이클은 완료가 아니라 `재실행 필요`로 판정한다.
