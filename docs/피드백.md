## 4. Phase 2: Pre-Response Timer Redesign

### 4.1 Problem Statement

**Error 1 — Widget lifecycle crash:**

```
Looking up a deactivated widget's ancestor is unsafe.
At this point the state of the widget's element tree is no longer stable.
To safely refer to a widget's ancestor in its dispose() method, save a
reference to the ancestor by calling dependOnInheritedWidgetOfExactType()
in the widget's didChangeDependencies() method.
```

**Error 2 — Timer logic must be redesigned** to use an additive modifier system.

### 4.2 Widget Lifecycle Fix

#### Root Cause

```dart
// ❌ WRONG — referencing context in dispose()
@override
void dispose() {
  final svc = context.read<SomeProvider>();   // deactivated widget
  Provider.of<Something>(context);            // ancestor lookup
  super.dispose();
}
```

#### Fix Pattern

```dart
// ✅ CORRECT — cache in didChangeDependencies, use in dispose
late final SomeType _cachedRef;

@override
void didChangeDependencies() {
  super.didChangeDependencies();
  _cachedRef = context.read<SomeProvider>();
}

@override
void dispose() {
  _cachedRef.cancel(); // safe
  super.dispose();
}
```

**Action:** grep the entire codebase for `context.read`, `context.watch`, `Provider.of`, and `dependOnInheritedWidgetOfExactType` inside any `dispose()` method. Fix every occurrence.

### 4.3 Additive Timer Formula

#### Variable Definitions

| Variable | Role | Default | Unit | Notes |
|----------|------|---------|------|-------|
| `base` | Base wait time | 30 | min | Always positive. User-configurable |
| `deviation` | Random jitter **range** | 10 | min | Each trigger: `random(-10, +10)` applied |
| `overlayOn` | Modifier when Live2D overlay is active | −20 | min | Applied only while overlay is visible |
| `screenLandscape` | Modifier when screen is in landscape | +20 | min | Applied only in landscape orientation |
| `screenOff` | Modifier when screen is off | `∞` | — | Means: **suspend timer entirely** |

#### Formula

```
if (screenOff)
    effectiveTime = ∞          // timer paused, no notification

else
    effectiveTime = base
                  + random(-deviation, +deviation)
                  + (overlayActive ? overlayOn : 0)
                  + (landscape    ? screenLandscape : 0)

Validation:
    if effectiveTime < 0 → CONFIG ERROR (reject & warn user)
    if effectiveTime = 0 → immediate trigger (allowed)
```

#### Worked Examples

```
Example 1 — Default state (no modifiers active):
  30 + rand(−10…+10) = 20–40 min  ✅

Example 2 — Overlay ON:
  30 + rand(−10…+10) + (−20) = 0–20 min  ✅
  ⚠ minimum is 0 → immediate trigger possible

Example 3 — Overlay ON + Landscape:
  30 + rand(−10…+10) + (−20) + (+20) = 20–40 min  ✅

Example 4 — Screen OFF:
  effectiveTime = ∞ → timer suspended  ✅

Example 5 — ERROR case (base=10, overlayOn=−20, deviation=10):
  worst case: 10 + (−10) + (−20) = −20  ❌ CONFIG ERROR
```

#### Minimum-Base Guard

To prevent config errors at save time:

```
minPossibleTime = base - abs(deviation) + sum(all negative modifiers that can be simultaneously active)

if (minPossibleTime < 0)
    → Block save, show warning:
      "Base time is too low for the current modifiers.
       Minimum base for this configuration: {abs(deviation) + abs(negativeModifiers)} min"
```

### 4.4 Screen-Off Behavior Detail

```
Screen turns OFF
  └─ Timer pauses (record elapsed time)

Screen turns ON
  └─ remainingTime = effectiveTime − elapsedBeforePause
  └─ if remainingTime ≤ 0 → trigger immediately
  └─ else → resume with remainingTime
  └─ (Do NOT recalculate random deviation — use original value)
```

| # | Task | Details |
|---|------|---------|
| 2-1 | Fix all `dispose()` context references | Grep + fix every instance project-wide |
| 2-2 | Implement `PreResponseTimer` with additive formula | Pure logic class, no UI dependency |
| 2-3 | Implement environment listeners | Overlay state, screen orientation, screen on/off — real-time listeners that feed into the timer |
| 2-4 | Implement timer recalculation on env change | When overlay/landscape state changes mid-timer: cancel current timer, recalculate, restart. Preserve `deviation` value within the same cycle |

## 7. Phase 5: Live2D–LLM Interaction

### 7.1 Overview

Embed Live2D control directives inside LLM responses so the character **expresses contextually appropriate emotions and motions in real time** during conversation.

### 7.2 Architecture

```
User Input
    │
    ▼
┌──────────────────────────────────────────┐
│ LLM API Call                             │
│ (system prompt includes available        │
│  motions/params + control syntax)        │
└──────────────────┬───────────────────────┘
                   │ raw response
                   ▼
┌──────────────────────────────────────────┐
│ Response Parser                          │
│                                          │
│ Extract control directives:              │
│   • Regex-based inline tags              │
│   • Lua script blocks                    │
│                                          │
│ Output:                                  │
│   ├─ Clean text (display in chat)        │
│   └─ Command queue (send to Live2D)      │
└──────────┬──────────────┬────────────────┘
           │              │
           ▼              ▼
     [Chat UI]     [Live2D Controller]
      display       execute motions /
      text          set parameters
```

### 7.3 Control Tag Syntax

#### Option A: Regex Inline Tags (simple cases)

```
LLM response example:
"Hi there! Nice to meet you. [motion:happy_wave] I'm in a great mood today!
[expression:happy] [param:ParamMouthOpenY=0.8,ParamEyeLOpen=1.0]"

Regex patterns:
  Motion:     \[motion:(\w+)\]
  Expression: \[expression:(\w+)\]
  Param:      \[param:([\w=.,]+)\]
```

#### Option B: Lua Script Blocks (complex sequences)

````
LLM response example:
"Hi there! Nice to meet you.

```lua-live2d
motion("happy_wave")
expression("happy")
param("ParamMouthOpenY", 0.8)
wait(2.0)
motion("idle_01")
```
"

Available Lua functions:
  motion(name)                        -- play a motion
  expression(name)                    -- set expression
  param(name, value)                  -- set parameter instantly
  param_lerp(name, target, duration)  -- interpolate parameter over time
  wait(seconds)                       -- pause sequence
  reset()                             -- reset all parameters to defaults
  preset(name)                        -- apply a preset from Phase 3
````

#### Option C: Hybrid (Recommended)

```
• Quick emotional cues → inline regex tags (natural mid-sentence)
• Complex choreography → Lua block (appended at end of response)
• Both can coexist in a single response
```

### 7.4 System Prompt Auto-Generation

```
[Pipeline]
Model loaded
  → parse .model3.json (reuse Phase 3 parser)
  → extract motion names, expression names, parameter names + ranges
  → inject into system prompt template

[Template excerpt]
"You are a Live2D character. During conversation, express emotions using
these tags:

- [motion:NAME]      — play a motion
- [expression:NAME]  — change expression
- For complex sequences, use a ```lua-live2d``` code block.

Available motions: idle_01, idle_02, happy_wave, sad_nod, ...
Available expressions: happy, sad, angry, neutral, surprised, ...
Available parameters: ParamAngleX (−30 to 30), ParamEyeLOpen (0 to 1), ..."
```

### 7.5 Response Parser Detail

```dart
class Live2DResponseParser {

  ParsedResponse parse(String raw) {
    final commands = <Live2DCommand>[];

    // 1) Extract and parse Lua blocks
    final luaPattern = RegExp(r'```lua-live2d\n(.*?)```', dotAll: true);
    for (final match in luaPattern.allMatches(raw)) {
      commands.addAll(_executeLuaSandbox(match.group(1)!));
    }

    // 2) Extract inline tags
    for (final m in RegExp(r'\[motion:(\w+)\]').allMatches(raw)) {
      commands.add(MotionCommand(m.group(1)!));
    }
    for (final m in RegExp(r'\[expression:(\w+)\]').allMatches(raw)) {
      commands.add(ExpressionCommand(m.group(1)!));
    }
    for (final m in RegExp(r'\[param:([\w=.,]+)\]').allMatches(raw)) {
      commands.addAll(_parseParamTag(m.group(1)!));
    }

    // 3) Strip all tags/blocks → clean display text
    final clean = raw
        .replaceAll(luaPattern, '')
        .replaceAll(RegExp(r'\[motion:\w+\]'), '')
        .replaceAll(RegExp(r'\[expression:\w+\]'), '')
        .replaceAll(RegExp(r'\[param:[\w=.,]+\]'), '')
        .trim();

    return ParsedResponse(displayText: clean, commands: commands);
  }
}
```

### 7.6 Lua Sandbox Constraints

```
[Security Rules]
├─ Only whitelisted functions exposed (motion, expression, param, etc.)
├─ No file I/O
├─ No network access
├─ No os/io/debug libraries
├─ Execution timeout: 5 seconds max
├─ Parameter values auto-clamped to model's min–max range
└─ Non-existent motion/param names → warn log + skip (no crash)
```

### 7.7 Streaming Response Handling

```
Problem: In streaming mode, tags may arrive split across chunks.
  e.g., chunk 1: "Hello [mot"   chunk 2: "ion:happy_wave] world"

Solution:
  • Maintain a look-ahead buffer
  • When '[' is detected, enter BUFFER mode
  • Accumulate until ']' is found or max tag length exceeded
  • On ']': parse complete tag → execute command, emit clean text
  • On max length exceeded: flush buffer as plain text (not a tag)
  • Lua blocks: buffer from '```lua-live2d' until closing '```'
```

### 9.5 Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| `dispose()` fix breaks other widgets | Phase 2 touches many files | List all affected widgets beforehand; fix and test one at a time |
| Removing persistent notification kills background execution | App gets killed by OS | Verify background keep-alive mechanism is independent of persistent notification before removal |
| No stable Lua interpreter for Flutter/Dart | Phase 5 blocker | Pre-evaluate at least 2 packages (`lua_dardo`, `flua`) before Phase 5 begins. Fallback: regex-only mode without Lua |
| Different Live2D models have different parameter structures | Phase 3 & 5 parser failures | Test parser against ≥3 structurally different models |
| Streaming response splits tags mid-chunk | Phase 5 parser corruption | Buffer-mode design in 7.7; fuzz-test with randomized chunk splits |
| Save/load schema changes between versions | Phase 4 data loss | Version field in schema + explicit migration functions |
