### Issue 3: 한글 깨짐 (Korean Text Corruption)
**기획반영.md 요구**: UTF-8 스트리밍 디코더 감사, 멀티바이트 버퍼 추가  
**현재 상태**: ❌ **미수행** — 오히려 **소스 파일 자체에 한글 깨짐 존재 확인**

**근거:**
- `api_service.dart`에서 `utf8` 검색 → **0건**. UTF-8 디코더 사용 안 함
- `chat_screen.dart` L678: `'硫붿떆吏 ??젣'` — 원래 '메시지 삭제'여야 하는 한글이 깨져있음
- `chat_screen.dart` L686: `'蹂듭궗'` — 원래 '복사'여야 하는 한글이 깨져있음

> ⚠️ **이것은 소스 파일의 인코딩 문제입니다.** 파일이 EUC-KR로 인코딩된 부분이 있거나, IDE에서 UTF-8이 아닌 인코딩으로 저장된 것으로 보입니다. 스트리밍 디코더 문제와 별도로, 소스 코드 자체의 한글도 깨져 있습니다.

**추가 발견:**
- 파일 내 다른 한글 문자열 (L852-853의 `'이미지 첨부'`, L872의 `'메시지를 입력하세요..'` 등)은 정상 → **일부 영역만 인코딩 손상**
- 구체적으로 L678 (`'硫붿떆吏 ??젣'`)와 L686 (`'蹂듭궗'`)는 EUC-KR→UTF-8 변환 과정에서 깨진 것으로 추정

**해결 필요 항목:**
1. 깨진 한글 문자열을 정상 한글로 수정 (`'硫붿떆吏 ??젣'` → `'메시지 삭제'`, `'蹂듭궗'` → `'복사'`)
2. API 스트리밍 응답에 `utf8.decoder` 적용 여부 확인
3. 전체 `.dart` 파일 인코딩 일관성 검증

**결론**: Issue 3 미수행. 추가로 소스 파일 자체의 인코딩 손상도 발견됨.

### 1. 소스 코드 인코딩 오염 발견

이는 기획반영.md의 Issue 3 (한글 깨짐)과 **별도의 문제**입니다:

- **새로운 발견**: 소스 파일 자체의 인코딩 손상

### 2. 네이티브 알림 코드 완성도 상향
기획반영.md에서는 알림 네이티브 코드가 없는 것으로 추정했으나,
실제로는 `notifications/` 폴더에 4개 파일 (NotificationHelper.kt, NotificationConstants.kt, NotificationActionReceiver.kt, NotificationActionStore.kt)이 **완전히 구현**되어 있습니다.

### 3. `supportsVision` 플래그 완전 구현 확인
PR Phase 1의 마지막 "선택적" 항목이었던 `supportsVision`이 `ApiConfig` 모델에 이미 완전 구현되어 있고, ChatScreen UI에도 연동되어 있습니다.
