### 🚨 Issue 1: Screenshot Feature — Architecture Misinterpretation + Native Crash

**Error encountered:**
```
PlatformException(CAPTURE_EXCEPTION, Media projections require a foreground service
of type ServiceInfo.FOREGROUND_SERVICE_TYPE_MEDIA_PROJECTION, null, null)
```

**Problem:** The screenshot feature itself is reasonably well-implemented at the code level, but **its integration into the app is fundamentally wrong**. The screenshot button is currently placed directly in the main chat input bar, which misrepresents the intended architecture.

**Correct Architecture:**
> ⚠️ The screenshot feature is meant to be a **standalone module / service** — it should NOT have its own button in the main chat UI. Instead, **other features call this module** when they need a screenshot, and the module returns the captured image as an attachment. Think of it as a utility service, not a user-facing button.

**Required Actions:**
- [ ] **Remove the screenshot button from the main chat input bar immediately**
- [ ] Refactor `ScreenCaptureService` into a pure module/service pattern (no direct UI trigger)
- [ ] Fix the `FOREGROUND_SERVICE_TYPE_MEDIA_PROJECTION` crash by adding the required foreground service type declaration in `AndroidManifest.xml` and updating the Kotlin native code
- [ ] Expose a clean API: `ScreenCaptureService.capture() → Future<ImageAttachment?>` that other features can call
- [ ] **Partial redesign order issued** — the feature's role in the app needs to be re-scoped

### 🚨 Issue 2: Image Display & Persistence — Images Not Shown After Sending

**Problem:** Users can attach and send images, but **images are not displayed in the chat history**. It appears that images are transmitted to the API but are **not persisted locally** — they vanish from the UI after sending.

**Required Architecture Changes:**

> ⚠️ This is a significant gap. Images must be **stored locally**, **displayed in chat**, **managed by the user**, and **included in past memory context** when relevant.

- [ ] **Image Cache Management Tab in Settings Menu**
  - Add a new settings section: "Image Cache Management"
  - Allow users to configure **maximum number of images displayed per chat session**
  - Provide **image compression settings** (quality, max resolution)
  - Show total cache size and provide a "Clear All Cache" button

- [ ] **Image Persistence**
  - Store images efficiently on local storage (not just in-memory base64)
  - Use a file-based cache with database references (path stored in message DB, file on disk)
  - **When a chat session is deleted, all associated images must be deleted** from storage
  - Users must also be able to **manually delete individual images** from chat history

- [ ] **Image Display in Chat**
  - Sent images must appear as inline thumbnails in the chat bubble
  - Tapping a thumbnail should open a full-size preview
  - Loading indicators for images that are still being processed

- [ ] **Past Memory Integration**
  - When images fall within the scope of past memory/context window, **they must be re-sent in the same context**
  - The past memory system must be aware of image attachments and include them when reconstructing conversation history for the API
  - This ensures the LLM maintains visual context continuity across conversation turns

### 🚨 Issue 3: Korean Text Corruption (한글 깨짐)

**Problem:** Korean characters are **garbled/corrupted** in two scenarios:
1. **When starting a new chat session** — the first few messages display broken Korean characters
2. **During streaming responses** — Korean text in the assistant's reply is corrupted while being streamed

> ⚠️ This is a critical UX bug for Korean-language users and likely indicates an encoding issue in the streaming decoder or text rendering pipeline.

**Likely Causes:**
- UTF-8 multi-byte characters being split across stream chunks
- Missing `utf8.decoder` in the streaming response handler
- Text widget not properly handling partial Unicode code points during streaming

**Required Actions:**
- [ ] Audit `ApiService` streaming response handling for proper UTF-8 decoding
- [ ] Ensure `utf8.decoder.bind(stream)` or equivalent is used instead of raw byte-to-string conversion
- [ ] Add a stream buffer that accumulates partial multi-byte characters before rendering
- [ ] Test with Korean, Japanese, Chinese, and emoji input to verify all multi-byte characters render correctly
- [ ] Verify that the chat database stores and retrieves Korean text without corruption

### 🚨 Issue 6: Master On/Off — Intent Clarification & Implementation Gap

**Intended Behavior (Clarified):**

> ⚠️ The Master On/Off switch is **NOT** meant to reset or modify any individual feature settings. It is a **global activation toggle** that works as follows:

| Master State | Behavior |
|-------------|----------|
| **OFF** | All features (Live2D overlay, notifications, proactive responses, etc.) are **deactivated** — they do not fire, do not show, do not run. **However, all their individual settings remain fully editable.** Users can freely configure Live2D display, notification preferences, timer values, etc. while Master is OFF. |
| **ON** | The system **reads the current state of each feature's individual toggle** and activates only those features that are individually enabled. Features that were set to disabled remain disabled. |

**Key Principle:** Master Off = "freeze execution, not configuration." Users should be able to set everything up with Master OFF, then flip Master ON to activate their configured setup.

**Current Implementation Gap:**
- The current implementation partially disables subsystems but **does not clearly separate "execution" from "configuration"**
- Settings UI does not clearly communicate that features can be edited while Master is OFF
- When Master is turned ON, it does not properly read and respect each feature's individual enabled/disabled state

**Required Actions:**
- [ ] Ensure `GlobalRuntimeProvider.setEnabled(true)` triggers a scan of all feature-level toggles and activates only those that are individually enabled
- [ ] Ensure `GlobalRuntimeProvider.setEnabled(false)` stops all execution but **does NOT modify any feature-level toggle values**
- [ ] Settings UI: when Master is OFF, show a banner like "⚠️ Master is OFF — changes will take effect when Master is turned ON" but keep all controls interactive
- [ ] Do NOT grey-out or disable settings controls when Master is OFF (this contradicts the intent)
- [ ] Add clear visual indicator on main screen showing Master ON/OFF state
