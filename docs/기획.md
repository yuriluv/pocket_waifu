#### Phase 4: Live2D Display Edit Mode Fix

**Current State**: 
- `display_settings_screen.dart` (19.4 KB) provides a functional edit-mode UI during a single session: users can drag to reposition, pinch to scale, and rotate the Live2D model.
- `display_config_store.dart` (4.4 KB) handles persistence via `SharedPreferences`.
- `display_config.dart` (12.9 KB) and `display_preset.dart` (8.7 KB) define the data models.

(b) The `load()` method is called **after** the overlay widget has already rendered its first frame with default values, causing a visible "jump" from default position to saved position. (c) If saved data is corrupted or empty, the app crashes instead of falling back to defaults.

## ⚠️ Key Risks & Mitigations

| Risk | Severity | Mitigation Strategy |
|------|----------|---------------------|
| `Map<String, String>` → `Map<String, dynamic>` migration breaks existing text-only message flow | 🔴 High | (a) Before starting, run `grep -rn "Map<String, String>" lib/` to catalog every occurrence. (b) Create a `getTextContent()` helper that abstracts over both formats. (c) Maintain backward compatibility: text-only messages continue to use `String` content, not a single-element array. (d) Run the full existing test suite after migration. (e) Manually test all chat flows (single message, conversation history, system prompt) before proceeding to F1-3. |
| `dispose()` context-reference fixes cascade into unintended side effects in other widgets | 🔴 High | (a) Before starting, list all widgets that use `context.read/watch/Provider.of` in `dispose()`. (b) Fix one widget at a time. (c) After each fix, run the app and navigate to/from that screen 5 times—verify no new errors. (d) Write a brief regression note for each fix documenting what was cached and where. |
| Removing persistent notification code breaks background execution / keep-alive | 🟡 Medium | (a) Before removing: document exactly how the app currently stays alive in the background. Is it solely via the foreground service, or are there other mechanisms (WorkManager, AlarmManager, boot receiver)? (b) If the foreground service is the only keep-alive mechanism, implement an alternative (e.g., `WorkManager` periodic task) before removing it. (c) After removal: verify the pre-response timer still fires when the app is backgrounded for 30+ minutes. |
| No stable, well-maintained Lua interpreter package exists for Flutter/Dart | 🟡 Medium | (a) Before starting Phase 5: evaluate at least 2 packages (`lua_dardo`, `flua`) with a minimal proof-of-concept (execute a simple script, call a bound Dart function from Lua, test timeout behavior). (b) Document findings. (c) If no package meets requirements (stability, security, timeout support), fall back to **Regex-only mode**: Phase 5 ships without Lua support, using only inline tag parsing. The Lua sandbox becomes a future enhancement. (d) The settings UI should still have the Lua toggle, but it shows "Coming soon" when disabled. |
| Different Live2D models have structurally different `.model3.json` schemas (different parameter names, motion group structures) | 🟡 Medium | (a) Test the `.model3.json` parser with at least 3 structurally different models: a standard Cubism sample model (e.g., Hiyori), a user-created model with custom parameters, and a model with minimal configuration. (b) The parser must not crash on any valid `.model3.json`. (c) Missing sections should result in empty lists, not exceptions. |
| Streaming LLM responses split inline tags across chunk boundaries (e.g., `"[mot"` + `"ion:Idle/0]"`) | 🟡 Medium | (a) Implement the buffer mode design from feedback.md §7.7 (enter buffer on `[`, accumulate until `]`, flush if max length exceeded). (b) Write a fuzz test that takes a known response, randomly splits it into chunks of 1–20 characters (100 iterations with different random seeds), processes each split through the streaming parser, and asserts that the final output (extracted tags + clean text) matches the expected result for every split pattern. |
| MediaProjection API is not available on Android emulators | 🟡 Medium | (a) All screen capture development must include physical device test sessions. (b) Schedule at least 2 dedicated test sessions on physical devices during Sprint 4. (c) In the CI pipeline, skip screen capture tests and mark them as "requires physical device". (d) The `ScreenCaptureService` should detect emulator environment and return `permissionStatus = unavailable` gracefully. |

## 📝 Coding Rules (Extracted from feedback.md)

These rules must be followed by all contributors for the duration of this project:

```
 1. Work on exactly ONE Phase at a time. Do not start Phase N+1 until Phase N
    passes all its verification scenarios.

 2. Within a Phase, work on exactly ONE Task at a time. Complete it fully
    (including any sub-items) before moving to the next Task.

 3. After completing each Task, run `flutter build apk --debug` and confirm
    zero errors. Run the app and smoke-test the affected screen(s).

 4. Removal/deletion tasks ALWAYS precede implementation tasks within the
    same Phase. Clean up broken code before writing new code.

 5. UI changes ALWAYS come after logic changes. Get the logic working and
    tested first, then build/update the UI on top of it.

 6. NEVER directly reference `context` (via `context.read`, `context.watch`,
    `Provider.of(context)`) inside a `dispose()` method. Cache the needed
    reference in `didChangeDependencies()` or `initState()` instead.

 7. Any arithmetic that could produce a negative or zero result (e.g., timer
    interval calculations) MUST include validation: block at save time,
    clamp/skip at runtime. Never allow a non-positive timer interval.

 8. All Live2D control commands originating from LLM output MUST execute
    within the Lua sandbox or through the validated tag parser. No raw,
    unvalidated strings should reach the Live2D controller.

 9. References to non-existent motions, expressions, or parameters MUST
    result in a warning log + skip. NEVER crash the app due to a missing
    asset reference.

10. All persisted settings MUST have a default-value fallback. If saved data
    is missing, corrupted, or schema-incompatible, the app must load
    gracefully with sensible defaults.

11. All new features MUST respect the Master Switch. When Master Switch is
    OFF, new features must either not initialize or must be in a dormant
    state. Test every new feature with Master Switch OFF to confirm it does
    not activate.
```
